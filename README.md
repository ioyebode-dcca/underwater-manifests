# Underwater Manifests (GitOps Repo)

This repository contains all Kubernetes manifests for the Underwater application. It is managed by Flux CD using the GitOps pattern — Flux watches this repo and automatically deploys any changes to the EKS cluster.

---

## How GitOps Works

```
Developer pushes code to EKS repo
         ↓
Jenkins pipeline builds and scans the image
         ↓
Jenkins updates deployment.yaml in THIS repo with new image tag
         ↓
Flux (running on EKS) detects the change
         ↓
Flux pulls the new image from ECR and deploys it to the cluster
         ↓
App is live — no manual kubectl apply needed
```

## Prerequisites

This repo is used alongside the EKS repo. Complete the EKS repo setup first:
- EKS cluster running (`terraform apply` in `EKS/dev`)
- kubectl connected to the cluster
- Flux CLI installed

## Step 1 — Add Deploy Key for Jenkins

Jenkins needs write access to push image tag updates:

```bash
# In WSL — generate SSH key (skip if already done in EKS repo setup)
ssh-keygen -t ed25519 -C "jenkins@underwater" -f ~/.ssh/jenkins_key -N ""

# Print public key
cat ~/.ssh/jenkins_key.pub
```

Add to GitHub:
```
github.com → underwater-manifests
→ Settings → Deploy Keys → Add deploy key

Title:  jenkins-deploy-key
Key:    paste contents of ~/.ssh/jenkins_key.pub
✅ Allow write access
→ Add key
```

## Step 3 — Bootstrap Flux on EKS Cluster

Run this after `terraform apply` has created the EKS cluster:

```bash
# Connect kubectl to dev cluster
aws eks update-kubeconfig \
  --region us-east-1 \
  --name underwater-dev

# Verify nodes are ready
kubectl get nodes

# Bootstrap Flux
flux bootstrap github \
  --owner=<your-github-username> \
  --repository=underwater-manifests \
  --branch=main \
  --path=clusters/dev \
  --personal

# Verify Flux pods are running
kubectl get pods -n flux-system
```

Expected output:
```
NAME                                      READY   STATUS
helm-controller-xxx                       1/1     Running
kustomize-controller-xxx                  1/1     Running
notification-controller-xxx               1/1     Running
source-controller-xxx                     1/1     Running
```

## How Jenkins Updates This Repo

Every time the Jenkins pipeline runs successfully it:

1. Clones this repo
2. Updates the image tag in `apps/base/<service>/deployment.yaml`
3. Commits and pushes to main branch

```bash
# Example of what Jenkins changes
# Before:
image: 495905914919.dkr.ecr.us-east-1.amazonaws.com/underwater:dev-12

# After build #13:
image: 495905914919.dkr.ecr.us-east-1.amazonaws.com/underwater:dev-13
```

Flux detects this commit within 1 minute and applies the new deployment automatically.

## Deployment YAML Structure

```yaml
# apps/base/underwater/deployment.yaml
apiVersion: apps/v1
kind: Deployment
metadata:
  name: ecr-app-underwater
  namespace: underwater
spec:
  replicas: 2
  selector:
    matchLabels:
      app: octopus-underwater-app
  strategy:
    type: RollingUpdate      # Zero downtime deployments
    rollingUpdate:
      maxSurge: 1            # One extra pod during update
      maxUnavailable: 0      # Never kill pod before new one is ready
  template:
    spec:
      containers:
        - name: octopus-underwater-app
          # Jenkins updates this tag on every build
          image: 495905914919.dkr.ecr.us-east-1.amazonaws.com/underwater:latest
          ports:
            - containerPort: 80
          readinessProbe:    # Pod only gets traffic when ready
            httpGet:
              path: /
              port: 80
          livenessProbe:     # Restart pod if unhealthy
            httpGet:
              path: /
              port: 80
```

## Kustomize Overlays Explained

Kustomize allows you to share base manifests and override values per environment:

```
Base (apps/base/underwater/)     ← shared config
         ↓
Dev overlay (apps/overlays/dev/) ← overrides: 1 replica, less resources
         ↓
Prod overlay (apps/overlays/prod/) ← overrides: 3 replicas, more resources
```

**Dev overlay example:**
```yaml
# apps/overlays/dev/kustomization.yaml
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - ../../base/underwater
patches:
  - target:
      kind: Deployment
      name: ecr-app-underwater
    patch: |
      - op: replace
        path: /spec/replicas
        value: 1
```

## Manual Deployment (Without Flux)

If needed, you can deploy manually:

```bash
# Apply base manifests
kubectl apply -k apps/base/underwater/

# Apply dev overlay
kubectl apply -k apps/overlays/dev/

# Check rollout
kubectl rollout status deployment/ecr-app-underwater -n underwater

# Rollback if needed
kubectl rollout undo deployment/ecr-app-underwater -n underwater
```

## Troubleshooting

**Flux not syncing:**
```bash
# Check Flux logs
flux logs --level=error

# Check if source is reachable
flux get sources git

# Force reconcile
flux reconcile source git flux-system
```

**Image pull error (ECR auth):**
```bash
# Check IRSA is configured correctly
kubectl describe pod <pod-name> -n underwater

# Verify node IAM role has ECR access
aws iam list-attached-role-policies \
  --role-name <node-role-name>
```

**Deployment stuck:**
```bash
# Check pod events
kubectl describe pod <pod-name> -n underwater

# Check pod logs
kubectl logs <pod-name> -n underwater

# Check deployment status
kubectl rollout status deployment/ecr-app-underwater -n underwater
```

**Nothing to commit error in Jenkins:**
```
This means the image tag in deployment.yaml already matches the current build.
The pipeline handles this gracefully with:
git diff --staged --quiet || git commit -m "..."
```

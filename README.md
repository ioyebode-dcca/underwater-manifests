# Underwater Manifests (GitOps Repo)

This repository contains all Kubernetes manifests for the Underwater application. It is managed by Flux CD using the GitOps pattern — Flux watches this repo and automatically deploys any changes to the EKS cluster.

---

## How GitOps Works

```
Developer pushes code to EKS repo
         ↓
Jenkins pipeline builds and scans the image
         ↓
Jenkins updates deployment.yaml in THIS repo with new image tag
         ↓
Flux (running on EKS) detects the change within 1 minute
         ↓
Flux pulls the new image from ECR and deploys it to the cluster
         ↓
App is live — no manual kubectl apply needed
```

## Step 1 — Add Deploy Key for Jenkins

Jenkins needs write access to update image tags in this repo:

```bash
# Generate SSH key in WSL
ssh-keygen -t ed25519 -C "jenkins@underwater" -f ~/.ssh/jenkins_key -N ""

# Print public key
cat ~/.ssh/jenkins_key.pub
```

Add to GitHub:
```
github.com → underwater-manifests
→ Settings → Deploy Keys → Add deploy key

Title:  jenkins-deploy-key
Key:    paste contents of ~/.ssh/jenkins_key.pub
✅ Allow write access
→ Add key
```

Test the key works:
```bash
ssh -i ~/.ssh/jenkins_key -T git@github.com
# Expected: Hi ioyebode-dcca/underwater-manifests! You've successfully authenticated
```

## Step 2 — Grant Nodes ECR Access

Node IAM role needs ECR read access to pull images:

```bash
# Get node group name
aws eks list-nodegroups \
  --cluster-name underwater-dev \
  --region us-east-1

# Get node role (replace nodegroup name with actual name from above)
NODE_ROLE=$(aws eks describe-nodegroup \
  --cluster-name underwater-dev \
  --nodegroup-name <nodegroup-name> \
  --query 'nodegroup.nodeRole' \
  --output text | cut -d'/' -f2)

echo "Node role: $NODE_ROLE"

# Attach ECR read policy
aws iam attach-role-policy \
  --role-name $NODE_ROLE \
  --policy-arn arn:aws:iam::aws:policy/AmazonEC2ContainerRegistryReadOnly

# Verify
aws iam list-attached-role-policies \
  --role-name $NODE_ROLE \
  --query 'AttachedPolicies[*].PolicyName' \
  --output table
```

## Step 3 — Bootstrap Flux on EKS Cluster

> ⚠️ Must include `--components-extra` flag for image automation to work

```bash
# Connect kubectl to dev cluster
aws eks update-kubeconfig \
  --region us-east-1 \
  --name underwater-dev

# Verify nodes are ready
kubectl get nodes

# Grant your IAM user cluster admin access (required first time)
aws eks create-access-entry \
  --cluster-name underwater-dev \
  --principal-arn arn:aws:iam::<account-id>:user/<iam-user> \
  --type STANDARD

aws eks associate-access-policy \
  --cluster-name underwater-dev \
  --principal-arn arn:aws:iam::<account-id>:user/<iam-user> \
  --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy \
  --access-scope type=cluster

# Bootstrap Flux with image automation controllers
flux bootstrap github \
  --owner=<your-github-username> \
  --repository=underwater-manifests \
  --branch=main \
  --path=clusters/dev \
  --personal \
  --components-extra=image-reflector-controller,image-automation-controller
```

Enter your GitHub personal access token when prompted.

Expected successful output:
```

✔ helm-controller: deployment ready
✔ kustomize-controller: deployment ready

✔ image-automation-controller: deployment ready
✔ all components are healthy

```

Verify all Flux pods are running:

## How Jenkins Updates This Repo

Every successful pipeline build:
1. Clones this repo via SSH
2. Updates the image tag in `apps/base/underwater/deployment.yaml`
3. Commits and pushes to main branch
4. Flux detects the commit within 1 minute and deploys automatically

```bash
# What Jenkins changes each build:
# Before:
image: 495905914919.dkr.ecr.us-east-1.amazonaws.com/underwater:dev-14

# After build #15:
image: 495905914919.dkr.ecr.us-east-1.amazonaws.com/underwater:dev-15
```

## Important Files Explained

### namespace.yaml
```yaml
# CRITICAL: Without this file Flux will delete the namespace on every sync
apiVersion: v1
kind: Namespace
metadata:
  name: underwater
```

### serviceaccount.yaml
```yaml
# Required by deployment.yaml → serviceAccountName: underwater-app
apiVersion: v1
kind: ServiceAccount
metadata:
  name: underwater-app
  namespace: underwater
```

### kustomization.yaml (base)
```yaml
# Lists ALL resources — if a file is not listed here Flux ignores it
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources:
  - namespace.yaml
  - serviceaccount.yaml
  - deployment.yaml
  - service.yaml
  - image-policy.yaml
```

### infrastructure/*/kustomization.yaml
```yaml
# CRITICAL: Every directory referenced by Flux MUST have this file
# even if empty — otherwise bootstrap fails with kustomization.yaml not found
apiVersion: kustomize.config.k8s.io/v1beta1
kind: Kustomization
resources: []
```

## Troubleshooting

**Bootstrap fails: kustomization.yaml not found**
```
Every directory referenced by Flux must have a kustomization.yaml file.
Fix: create kustomization.yaml with resources: [] in empty directories
```

**Bootstrap fails: ImagePolicy not found**
```
Image automation controllers not installed.
Fix: add --components-extra=image-reflector-controller,image-automation-controller
     to the flux bootstrap command
```

**Namespace keeps getting deleted**
```
Flux only keeps what is defined in git.
Fix: add namespace.yaml to apps/base/underwater/ and list it in kustomization.yaml
```

**Pod fails: serviceaccount not found**
```
The deployment references serviceAccountName: underwater-app
Fix: add serviceaccount.yaml and list it in kustomization.yaml
```

**Pod fails: ImagePullBackOff**
```
Two possible causes:

1. Image tag does not exist in ECR (using :latest when only :dev-15 exists)
   Fix: check tags with:
        aws ecr list-images --repository-name underwater --region us-east-1
        Update deployment.yaml to use the exact tag

2. Node IAM role missing ECR permissions
   Fix: attach AmazonEC2ContainerRegistryReadOnly to node role (see Step 2)
```

**Pod fails: nginx chown Operation not permitted**
```
Deployment has overly restrictive securityContext dropping all capabilities.
Fix: remove the securityContext.capabilities block from the container spec
```

**Git push rejected: fetch first**
```
Flux or Jenkins pushed to the repo while you were working.
Fix: git pull origin main --rebase
     git push origin main
```

**Flux reconcile: context deadline exceeded**
```
Kustomize controller is stuck applying resources.
Fix: kubectl describe kustomization flux-system -n flux-system
     kubectl logs -n flux-system deployment/kustomize-controller --tail=30
```

**kubectl: server has asked for credentials**
```
IAM user not added to cluster access entries.
Fix:
  aws eks create-access-entry \
    --cluster-name underwater-dev \
    --principal-arn arn:aws:iam::<account-id>:user/<iam-user> \
    --type STANDARD

  aws eks associate-access-policy \
    --cluster-name underwater-dev \
    --principal-arn arn:aws:iam::<account-id>:user/<iam-user> \
    --policy-arn arn:aws:eks::aws:cluster-access-policy/AmazonEKSClusterAdminPolicy \
    --access-scope type=cluster
```

```
main    → Flux watches this for both dev and prod clusters

develop → Not watched by Flux
          Used for manifests development only
```
